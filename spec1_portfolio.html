01 — What This Is
Real-time signal monitoring for national security and geopolitical risk

SPEC-1 is an automated open-source intelligence engine. It harvests signals from authoritative sources, filters them through a deterministic four-gate framework, and surfaces what matters — producing structured intelligence records and daily written briefs without manual intervention.

One purpose: separate signal from noise in a high-volume information environment, and deliver what actually requires human attention.

02 — The Problem It Solves
The bottleneck is attention, not access

Anyone working in national security, geopolitical risk, or threat intelligence faces the same daily reality: there is far more credible reporting than any human can process. Feeds from think tanks, investigative journalists, and defense research organizations produce dozens of new pieces every day. Most are not actionable. A small fraction are. The challenge is not access to information — it is triage.

SPEC-1 automates that triage. It monitors a curated set of high-credibility sources continuously, evaluates each new signal through a structured scoring framework, and surfaces only those that clear every threshold. Signals that pass are handed off to an investigation and verification workflow powered by Claude. The result is a compressed, high-confidence set of intelligence records and a daily written brief — ready for human review without the noise.

03 — System Architecture
Seven sequential stages. Each transforms the data and gates what moves forward.

The pipeline runs end-to-end on a daily cron schedule and on demand via API. Every stage has a single responsibility. Failures at any stage are logged and skipped — the cycle never halts on a single error.

Stage Descriptions

01 — Harvest. A harvester module fetches RSS/Atom feeds from a curated list of national security sources — think tanks, investigative outlets, and defense policy organizations with consistent editorial standards. The harvester handles SSL edge cases, malformed XML, and timeout conditions gracefully. A failed feed does not stop the cycle — it is logged and skipped. Output: raw Signal objects carrying source, URL, raw text, author, timestamp, and engagement metadata.

02 — Parse. Each raw signal passes through a parser that strips HTML markup, extracts clean prose, identifies keywords and named entities, and measures content volume. BeautifulSoup handles HTML cleaning; lightweight NLP heuristics handle extraction — no external model dependencies. Output: ParsedSignal with normalized text, keyword set, entity set, word count, and detected language.

03 — Score (Four-Gate Framework). The core filtering layer. Every parsed signal must clear four independent gates. Any single failure removes the signal from further processing. Gates evaluate credibility, volume, velocity, and novelty. Signals clearing all four gates become Opportunity objects with a composite score blending the four gate dimensions. Specific thresholds, weights, keyword taxonomy, and time-window parameters are not published here — they represent the accumulated operational judgment of the system.

04 — Investigate. Each opportunity is handed to an investigation generator that constructs a structured Investigation object. This includes: a stated hypothesis, research queries targeting authoritative external sources, and a list of analyst leads. Analyst selection is domain-aware — signals about Russian military operations surface different leads than signals about cyber operations or energy infrastructure.

05 — Verify. Each investigation is submitted to Claude (Haiku model) for hypothesis verification. Claude evaluates the hypothesis and returns a structured classification with confidence score. If the API call fails for any reason, the system falls back gracefully — the cycle continues and the failure is logged. The pipeline never crashes on a verification error.

06 — Analyze. The analyzer synthesizes signal, investigation, and verification outcome into a final IntelligenceRecord. This stage blends confidence signals from multiple sources — verification confidence, source credibility, analyst credibility, outcome classification — into a single composite confidence value. The weighting of blended inputs reflects the relative epistemic weight each source carries. These weights are calibrated, not arbitrary.

07 — Store. Completed IntelligenceRecord objects are written via two paths. Append-only JSONL with thread-safe single-writer access — records are never overwritten, the store is the immutable operational log. SQLite receives a parallel write via cls_db.dual_write for structured querying, downstream analysis, and historical pattern access. JSONL is ground truth; SQLite is the query layer.

04 — Domain Adapters
Specialized adapters inside cls_osint

The core pipeline is domain-agnostic. Domain-specific intelligence is handled by adapter modules under cls_osint/adapters/ that feed into the same scoring and persistence infrastructure.

cls_osint/adapters/fara
FARA bulk filing ingestion. Ingests DOJ public database filings, cross-references foreign agent registrations against congressional activity and known influence networks. Surfaces registrant-principal relationships as scored intelligence signals.
22 tests
cls_osint/adapters/congressional
Congressional trade intelligence. Ingests member financial disclosures via a fallback chain: Quiver Quantitative → Capitol Trades → House eFD. Flags trades in defense, cybersecurity, and energy sectors for conflict-of-interest analysis.
35 tests
cls_osint/adapters/narrative
Psychological operations detection. Monitors tracked sources for coordinated language (TF-IDF cosine similarity clustering), entity mention anomalies, publication silence patterns, and campaign-level aggregation. Surfaces narrative seeding, astroturfing, source discrediting, and information suppression as scored signals. Outputs AnomalyRecord, SimilarityCluster, and CampaignRecord objects.
cls_db.dual_write
Atomic dual-write layer. Every store write is wrapped so JSONL and SQLite stay in sync. JSONL is the immutable ground truth; SQLite is the query layer for historical analysis and downstream access.
05 — Domain Adapters: Extended Coverage
Specialized modules extend the core pipeline into adjacent intelligence domains.

The Notitia Civica adapter monitors Portland Metro civic sources — city council, Multnomah County, TriMet, and PDX 911 — applying the same four-gate framework to local government signals. The Research Mode adapter runs analyst-defined topic dossiers: the analyst specifies a topic, the system expands it into a structured feed set, runs the full pipeline cycle, and produces a research brief with notable findings and collection gaps.

Note: the quantitative market intelligence module (cls_quant) was removed in v0.6.0. Defense and cybersecurity equity signals are no longer part of the core pipeline. Scope assessment determined the core triage mission is text-based OSINT; market signal demand was not validated for operational use.

06 — Daily Intelligence Brief
Claude Sonnet authors the brief. Rule-based fallback if the API is unavailable.

After each cycle, a briefing generator submits scored intelligence records to Claude Sonnet with a professional editor prompt — precise, sourced, no speculation beyond the evidence. Every claim traces back to a scored signal. If the API call fails for any reason, a structured fallback brief is generated from the raw record data. The cycle never produces nothing.

  • Executive summary of cycle findings
  • Elevated signals requiring immediate attention
  • Domain briefings — cyber and geopolitical separately
  • Story leads for further investigation
  • Watch list items for ongoing monitoring
07 — API and Scheduling
FastAPI service. APScheduler. Daily cron at 06:00 AM PT.

A FastAPI application wraps the full pipeline for operational use. The scheduler runs on a fixed daily cadence. A kill-file mechanism lets operators pause scheduled execution without touching configuration — no restart required.

  • Trigger an immediate pipeline cycle
  • Query cycle status and statistics
  • Retrieve recent signals, intelligence records, and briefs
  • Kill-switch control for scheduled execution

An MCP server (mcp_server.py) exposes the full pipeline surface to Claude directly: run_cycle, get_signals, get_intel, get_leads, get_brief, get_psyop, get_fara, analyse_psyop, get_stats, file_verdict, get_verdicts, get_calibration.

08 — How the System Learns and Improves
Explicit, legible, deterministic. The system learns — without learning implicitly.

SPEC-1 has no neural network, no gradient descent, no embedding space. The scoring framework is rule-based and deterministic. Improvement happens through calibration: thresholds are adjusted by humans, based on evidence surfaced by the system. When something is wrong, the specific failure is diagnosable and fixable.

Calibration as Learning

The four-gate framework and composite scoring weights were developed through iterative exposure to the signal environment. When a threshold is wrong, the failure is diagnosable. The specific threshold that failed can be identified and adjusted. A neural network learns implicitly. SPEC-1 learns explicitly.

Source Credibility as a Dynamic Signal

Source credibility ratings are not static. They reflect a judgment about editorial standards, domain expertise, and historical accuracy. A source that degrades in quality can be downweighted. A new authoritative source can be added with an initial credibility assessment.

Analyst Registry as Accumulated Knowledge

The analyst registry encodes domain expertise — which researchers and journalists have demonstrated sustained accuracy in specific subject areas. Connecting signal domains to specific analyst leads is itself a form of learned judgment.

Outcome Feedback Loop (Structural)

The verification stage produces outcome classifications with confidence scores. Over time, patterns in this data — sources that consistently produce corroborated signals, domains where the pipeline performs well or poorly — are legible from the JSONL records and the SQLite query layer. The system does not automate this feedback loop, but the data structure supports it.

Human verdicts are recorded via cls_verdicts. Calibration drift is surfaced descriptively by cls_calibration — it never auto-applies changes. Tuning is a human decision. This preserves the deterministic, legible design philosophy.

Test Suite as Learning Anchor

621 tests across 61 files. When a threshold or weight changes during calibration, the tests verify that the change does not break intended behavior elsewhere. The test suite is, in a real sense, a changelog of accumulated judgment.

09 — What Is Not Here
Operational parameters are not published here.
  • Gate thresholds and cutoff values
  • Composite scoring weights and their derivation
  • Keyword taxonomy used for novelty detection
  • Source credibility ratings
  • Analyst credibility scores and weighting scheme

These parameters represent the accumulated operational judgment of the system. They are not appropriate for public documentation.

11 — Why This Matters
The system handles the volume. The analyst handles the judgment.

The bottleneck in intelligence work is not information — it is attention. SPEC-1 is designed to protect analyst attention by doing the triage work that does not require human judgment: fetching, cleaning, filtering, and structuring.

What reaches the human is already scored, investigated, verified, and written up. The human role is then what it should be: evaluating the findings, directing follow-on research, and making decisions.

The system handles the volume. The analyst handles the judgment. That division of labor is the point.
